# 数据流架构说明

本文档说明本项目为什么需要 Flume、Kafka、Storm，以及它们在系统中的位置。B 同学开发大数据模块时必须参考本文档，C 同学展示页面中的“大数据流程状态”也应参考本文档。

## 1. 总体数据流

```text
hotel_bookings.csv
    ↓
Flume 数据采集
    ↓
Kafka 消息队列
    ↓
Storm 实时处理
    ├── 数据清洗 Bolt
    ├── 模型预测 Bolt
    └── 实时统计 Bolt
    ↓
MySQL 数据库存储
    ↓
Flask API 后端接口
    ↓
Web 前端可视化页面
```

## 2. 各组件职责

| 组件 | 作用 | 本项目中的任务 |
|---|---|---|
| CSV 数据源 | 原始数据 | 使用 `hotel_bookings.csv` 模拟酒店预订数据持续产生 |
| Flume | 数据采集 | 读取 CSV 文件或日志文件，将数据发送到 Kafka |
| Kafka | 消息队列 | 缓冲预订数据，作为 Flume 和 Storm 的中间层 |
| Storm | 实时处理 | 接收 Kafka 数据，完成清洗、预测、统计 |
| MySQL | 数据存储 | 保存清洗数据、预测结果、统计结果 |
| Flask | 后端服务 | 提供 dashboard、predict、analysis 等 API |
| Web 前端 | 可视化展示 | 展示统计图表、预测结果和系统流程 |

## 3. 为什么本项目适合大数据流处理

酒店预订在真实业务中是不断产生的，每一条预订都可以看作一条实时数据流。本项目虽然使用 Kaggle CSV 数据集，但可以通过逐行读取 CSV 来模拟实时预订数据进入系统。

这与课程要求的 Flume + Kafka + Storm 架构相匹配：

1. Flume 负责采集预订数据。
2. Kafka 负责缓存和分发数据。
3. Storm 负责实时清洗、预测和统计。
4. MySQL 保存处理结果。
5. Flask 和前端展示最终分析结果。

## 4. Storm 拓扑设计

Storm 中建议设计 3 个 Bolt：

### 4.1 数据清洗 Bolt

文件建议：`05_bigdata/storm/bolt_clean.py`

职责：
- 解析 Kafka 中的一行预订数据
- 处理缺失值
- 转换字段类型
- 删除或标记异常值
- 输出标准化后的预订数据

### 4.2 模型预测 Bolt

文件建议：`05_bigdata/storm/bolt_predict.py`

职责：
- 加载 `02_model/cancellation_model.pkl`
- 将清洗后的数据转换为模型输入格式
- 输出取消预测结果
- 生成 `cancel_probability`、`risk_level` 等字段

### 4.3 实时统计 Bolt

文件建议：`05_bigdata/storm/bolt_stat.py`

职责：
- 统计实时订单数量
- 统计实时取消率
- 统计不同酒店类型、客户类型、国家/地区的订单分布
- 将统计结果写入 MySQL 或中间结果文件

## 5. 项目中可以采用的实现方式

考虑到课程设计时间有限、组员基础较弱，本项目建议采用“两层实现”：

### 第一层：文档和目录上完整体现大数据架构

必须包含：
- `05_bigdata/flume/flume_hotel_booking.conf`
- `05_bigdata/kafka/producer_simulator.py`
- `05_bigdata/storm/topology.py`
- `05_bigdata/storm/bolt_clean.py`
- `05_bigdata/storm/bolt_predict.py`
- `05_bigdata/storm/bolt_stat.py`

### 第二层：代码中优先完成可演示的数据流模拟

如果真实 Flume/Kafka/Storm 配置困难，可以先用 Python 脚本模拟：

```text
run_pipeline_demo.py
    ↓
逐行读取 hotel_bookings.csv
    ↓
模拟发送到 Kafka
    ↓
调用清洗函数
    ↓
调用模型预测函数
    ↓
保存结果到 MySQL 或 CSV
```

注意：报告和答辩中要说明这是“课程设计环境下的单机模拟流程”，架构设计仍按照 Flume + Kafka + Storm。
