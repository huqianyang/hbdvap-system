# 开发时间表

本时间表按照三人小组、零基础、主要依赖 AI 工具开发的情况设计。实际开发中可以根据课程截止时间压缩或延长。

## 1. 总体阶段

| 阶段 | 时间 | 核心目标 | 负责人 |
|---|---|---|---|
| 第 1 阶段 | 第 1-2 天 | 统一项目规则、目录、字段、接口 | A |
| 第 2 阶段 | 第 3-5 天 | 数据清洗、特征工程、模型训练 | A |
| 第 3 阶段 | 第 5-8 天 | Flask 后端、数据库、模型调用、大数据模块 | B |
| 第 4 阶段 | 第 8-10 天 | 前端页面、图表、预测表单 | C |
| 第 5 阶段 | 第 10-12 天 | 前后端联调、模型接口联调、大数据流程演示 | A/B/C |
| 第 6 阶段 | 第 13-14 天 | 报告、PPT、截图、最终打包 | A/B/C |

## 2. 第 1 阶段：项目启动与规则统一

负责人：A

任务：
1. 创建项目目录结构。
2. 整理 `项目总说明.md`。
3. 整理 `项目目录结构.md`。
4. 整理 `字段规范.md`。
5. 整理 `接口文档.md`。
6. 整理 B/C 的 AI 提示词。
7. 把启动包发给 B/C。

交付物：
- 启动包文档
- 数据集
- 统一目录结构
- B/C 的任务说明和 AI 提示词

## 3. 第 2 阶段：数据与模型

负责人：A

任务：
1. 读取 `hotel_bookings.csv`。
2. 分析缺失值、异常值、字段类型。
3. 清洗数据，输出 `hotel_bookings_clean.csv`。
4. 做特征工程，输出 `model_features.csv`。
5. 训练取消预测模型。
6. 评估模型准确率、召回率、F1 值等指标。
7. 导出 `cancellation_model.pkl`。
8. 编写 `predict_demo.py` 给 B 调用模型。

交付物：
- `01_data/processed/hotel_bookings_clean.csv`
- `01_data/processed/model_features.csv`
- `02_model/cancellation_model.pkl`
- `02_model/feature_columns.json`
- `02_model/model_metrics.json`
- `02_model/predict_demo.py`

## 4. 第 3 阶段：后端与大数据模块

负责人：B

任务：
1. 搭建 Flask 项目。
2. 创建 MySQL 数据库表。
3. 导入清洗后数据。
4. 编写 `/api/dashboard`。
5. 编写 `/api/predict`，调用 A 的模型。
6. 编写多维分析接口。
7. 编写 `/api/bigdata/status`。
8. 完成 Flume/Kafka/Storm 目录、配置和模拟脚本。

交付物：
- `03_backend/app.py`
- `03_backend/database/schema.sql`
- `03_backend/database/import_data.py`
- `03_backend/api/*.py`
- `03_backend/utils/model_loader.py`
- `05_bigdata/*`

## 6. 第 5 阶段：联调测试

负责人：A/B/C

测试流程：
1. A 测试模型能否单独运行。
2. B 测试 Flask 接口能否返回 JSON。
3. B 测试 `/api/predict` 能否加载模型并返回预测结果。
4. C 测试前端页面能否请求 API。
5. 全员测试完整流程：输入预订信息 → 点击预测 → 展示预测结果 → 查看图表。
6. 测试大数据流程演示脚本。

交付物：
- 运行截图
- 接口测试截图
- 前端页面截图
- 大数据流程说明截图

## 8. 每日同步规则

每天晚上同步一次：

```text
1. 今天完成了什么？
2. 遇到什么问题？
3. 明天准备做什么？
4. 有没有改字段、接口、目录？
```

如果改字段或接口，必须先通知全员，不能直接改代码。

## 9. 最小可交付版本

如果时间不够，优先保证以下内容完成：

1. A：清洗数据 + 模型训练 + 模型文件
2. B：Flask 接口 + 预测接口 + 看板接口
3. C：首页 + 预测页面 + 2-3 个图表
4. 大数据模块：目录完整 + 架构说明 + 模拟流程脚本
5. 报告：说明 Flume/Kafka/Storm 架构设计
